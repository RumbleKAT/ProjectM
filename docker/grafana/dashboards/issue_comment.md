## 피드백 반영 및 대안 제안

사용자 유입 및 무분별한 요청에 의해 워크스페이스가 과도하게 생성되어 DB 및 벡터 DB에 부하를 주는 문제를 방지하고, 동시성 문제를 해결하기 위해 다음과 같이 설계를 보완합니다.

### 1. 단일 공유 워크스페이스 (`temp-api`) 사용 및 스레드 격리
- **워크스페이스 무분별 생성 방지**: 매 요청마다 새로운 워크스페이스를 만들지 않고, 시스템에 단 **하나**의 공유 임시 워크스페이스 (`temp-api`)를 생성하여 모든 임시 요청이 이를 공유하도록 합니다.
- **채팅 이력 격리**: 클라이언트의 세션을 격리하기 위해, 공유 워크스페이스 내에서 개별 요청 시 마다 **스레드(Thread)**를 임시로 생성하거나 `api_session_id`를 사용하여 채팅 이력을 논리적으로 격리합니다.
- **가벼운 리소스 관리**: 벡터 DB의 네임스페이스를 매번 생성/삭제하는 무거운 작업 대신, DB에서 24시간이 지난 임시 스레드/대화 기록만 삭제하므로 리소스 관리가 매우 가볍고 안전합니다.

### 2. 동시성 제어 및 대기열 (Queue) 도입
- **LLM 호출 제한 (Rate Limit 방지)**: 프로젝트에 이미 설치되어 있는 `p-queue` 라이브러리를 사용하여 동시 LLM 호출 제한 큐(예: 최대 동시성 5개)를 구축합니다.
- **동작 방식**:
  - **Sync 요청**: 동시성 제한에 도달하면 큐에서 차례를 기다린 뒤 완료되면 응답을 반환합니다. (대기가 너무 길어질 경우를 대비해 타임아웃 처리 가능)
  - **Async 요청**: 호출 시 즉시 `chatId`와 `pending` 상태를 응답하고, 백그라운드에서 대기열에 들어가 순차적으로 LLM 호출을 처리하여 완료되면 DB 값을 업데이트합니다.
  - 이로써 다수의 API 요청이 동시에 유입되어도 LLM API의 Rate Limit에 걸리거나 서버가 다운되는 현상을 방지합니다.
